Version Control System (VCS) কী?

VCS (ভার্সন কন্ট্রোল সিস্টেম) হলো এমন একটি সিস্টেম যা আমাদের ফাইলের পরিবর্তনগুলোকে সময় অনুযায়ী ট্র‍্যাক করে বা রেকর্ড করে রাখে। এর ফলে কোনো ফাইলে কী পরিবর্তন হয়েছে, কে করেছে, কেন করেছে—সবকিছুর ইতিহাস (History) সংরক্ষিত থাকে।

V1 প্রথম ড্রাফট
V2 বাগ ফিক্স
V3 ডিজাইন আপডেট History
V4 ফাইনাল কোড

VCS কেন দরকার? (Why do we need it?)

  • পূর্বের Version-এ ফিরে যাওয়া যায়: কোডে কোনো বড় ভুল হলে আগের যেকোনো ভালো ভার্সনে সহজেই ফিরে যাওয়া সম্ভব।
  • সহজ টিমওয়ার্ক (Collaboration): একাধিক টিম মেম্বার একই ফাইলের আলাদা আলাদা অংশে কোনো ঝামেলা ছাড়াই একসাথে কাজ করতে পারেন।
  • কোড নিরাপদ থাকে (Code Safety): ভুলবশত কোড ডিলেট বা নষ্ট হয়ে গেলেও গিট হিস্ট্রি থেকে সম্পূর্ণ কোড ফিরিয়ে আনা যায়।

মনে রাখুন:

গিট শিখলে আপনার কোডের ইতিহাস বা হিস্ট্রি কখনো হারাবে না! আপনি আত্মবিশ্বাসের সাথে কোড এডিট করতে পারবেন।

টপিক ০২

Git বনাম GitHub

অনেকেই গিট (Git) এবং গিটহাব (GitHub) কে একই জিনিস মনে করেন, কিন্তু তাদের মধ্যে স্পষ্ট পার্থক্য রয়েছে। সংক্ষেপে বলতে গেলে:

Git (গিট)

  • এটি একটি সফটওয়্যার (Software)।
  • এটি লোকাল কম্পিউটারেই কাজ করে (Local System)।
  • এটি ফাইল ট্র্যাকিং এবং ভার্সন কন্ট্রোল করে।
  • ইন্টারনেট ছাড়াও সম্পূর্ণ অফলাইনে কাজ করা যায়।
  • কোড আপনার নিজের কম্পিউটারে জমা থাকে।
VS

GitHub (গিটহাব)

  • এটি একটি ওয়েব সার্ভিস (Web Service)।
  • এটি একটি ক্লাউড বা ইন্টারনেট ভিত্তিক হোস্টিং সার্ভিস (Cloud)।
  • এটি গিট রিপোজিটরিগুলো অনলাইনে হোস্ট করতে সাহায্য করে।
  • এটি ব্যবহার করার জন্য ইন্টারনেট কানেকশন প্রয়োজন।
  • কোড ইন্টারনেটে ক্লাউড স্টোরেজে জমা থাকে।

গোল্ডেন রুল:

Git হলো আপনার লোকাল কাজের টুল, আর GitHub হলো আপনার করা কাজ অন্য সবার সাথে শেয়ার ও ব্যাকআপ রাখার ক্লাউড প্ল্যাটফর্ম।

টপিক ০৩

Git-এর ৩টি স্টেজ (3 Stages of Git)

গিটে আমাদের কাজ মূলত তিনটি প্রধান ধাপে বা স্টেজে সম্পন্ন হয়। এই ধাপগুলো বুঝতে পারলে গিটের পুরো প্রসেস বোঝা খুবই সহজ হয়ে যায়।

১

Working Directory

যেখানে আমরা ফাইল তৈরি বা এডিট করি

git add
২

Staging Area

ফাইলগুলোকে কমিট করার জন্য প্রস্তুত করা (Index)

git commit
৩

Git Repository

ফাইলগুলো স্থায়ীভাবে কমিট ও সেভ হয়ে যায় (.git)

সংক্ষেপে মনে রাখার টিপস:

1
git add: ফাইলগুলোকে Working Directory থেকে Staging Area-তে পাঠায়।
2
git commit: ফাইলগুলোকে Staging Area থেকে Local Repository-তে চিরস্থায়ীভাবে সেভ (snapshot) করে।
3
git push: লোকাল রিপোজিটরি থেকে কোডগুলো ক্লাউডে বা GitHub-এ আপলোড করে দেয়।

কাজের পুরো প্রবাহ (Workflow):

Working Directory → Staging Area → Git Repository (.git) → GitHub (Remote)

টপিক ০৪

প্রথমবার GitHub-এ কোড পুশ (Setup Steps)

প্রথমবার কোনো প্রজেক্ট তৈরি করে সেটি গিটহাবে আপলোড করার জন্য নিচের ৯টি সহজ ধাপ অনুসরণ করতে হয়।

১
প্রজেক্ট ফোল্ডার তৈরি করুন:

আপনার কম্পিউটারে একটি নতুন ফোল্ডার তৈরি করুন।

Terminal 
mkdir my-project
২
VS Code দিয়ে ফোল্ডারটি ওপেন করুন:

তৈরি করা ফোল্ডারটি আপনার কোড এডিটরে ওপেন করুন।

৩
গিট ইনিশিয়েট (Git Init) করুন:

প্রজেক্টে গিট চালু করার জন্য নিচের কমান্ডটি রান করুন। এর ফলে ফোল্ডারে একটি ইনভিজিবল .git ফোল্ডার তৈরি হবে।

Terminal 
git init
৪
ফাইলসমূহ Staging-এ যুক্ত করুন:

ফোল্ডারের সব ফাইল ট্র্যাকিং তালিকায় যুক্ত করতে . (ডট) ব্যবহার করে কমান্ডটি লিখুন।

Terminal 
git add .
৫
প্রথম কমিট (Commit) করুন:

সব ফাইল লোকাল রিপোজিটরিতে সেভ করার জন্য একটি অর্থবহ মেসেজ দিয়ে কমিট করুন।

Terminal 
git commit -m "first commit"
৬
ডিফল্ট ব্রাঞ্চ 'main' নির্ধারণ করুন:

আপনার ডিফল্ট কাজের ব্রাঞ্চের নাম main সেট করুন।

Terminal 
git branch -M main
৭
GitHub-এ একটি রিপোজিটরি তৈরি করে URL কপি করুন:

গিটহাব অ্যাকাউন্টে গিয়ে একটি নতুন Repository তৈরি করুন এবং তার HTTPS বা SSH লিঙ্কটি কপি করে আনুন।

৮
রিমোট অরিজিন (Remote Origin) সেট করুন:

কপি করা গিটহাব লিঙ্কটি আপনার লোকাল প্রোজেক্টের সাথে যুক্ত করার জন্য নিচের কমান্ডটি দিন।

Terminal 
git remote add origin https://github.com/username/repo-name.git
৯
কোড পুশ (Push) করুন:

লোকাল প্রজেক্টের সব কোড গিটহাবে পাঠানোর জন্য নিচের কমান্ডটি রান করুন।

Terminal 
git push -u origin main
টপিক ০৫

পরবর্তী সময়ে পরিবর্তন পুশ করার নিয়ম

একবার প্রজেক্টের সাথে GitHub কানেক্ট হয়ে গেলে, পরবর্তীতে ফাইলে নতুন কোনো কোড পরিবর্তন বা সংযোজন করার পর গিটহাবে পুশ করার প্রসেসটি অত্যন্ত সংক্ষিপ্ত। তখন আর বার বার ইনিশিয়ালাইজ বা রিমোট লিঙ্ক যোগ করতে হয় না।

মাত্র ৩টি সহজ কমান্ড:

কমান্ড ১

পরিবর্তনগুলো যুক্ত করুন

সব নতুন বা পরিবর্তিত ফাইল স্টেজিং এরিয়াতে যোগ করার কমান্ড:

Terminal 
git add .
কমান্ড ২

কমিট করুন

পরিবর্তনের একটি ছোট বিবরণ দিয়ে স্ন্যাপশট সেভ করার কমান্ড:

Terminal 
git commit -m "update message"
কমান্ড ৩

সরাসরি পুশ করুন

নতুন কমিটগুলো সরাসরি রিমোট রিপোজিটরিতে পাঠিয়ে দেয়ার কমান্ড:

Terminal 
git push

কাজ শেষ!

এই ৩টি কমান্ড রান করার সাথে সাথেই আপনার পরিবর্তিত কোড গিটহাবে নিরাপদে আপডেট হয়ে যাবে।

টপিক ০৬

গুরুত্বপূর্ণ কমান্ডসমূহ (Git Commands Cheat Sheet)

নিচে দৈনন্দিন কাজে সবচেয়ে বেশি ব্যবহৃত গিট কমান্ডগুলোর একটি তালিকা দেওয়া হলো। এই কমান্ডগুলো ভালোভাবে অনুশীলন করলে কাজ অনেক সহজ হয়ে যাবে।

কমান্ড (Command) ব্যবহার (Usage) উদাহরণ (Example)
git init নতুন লোকাল রিপোজিটরি শুরু করতে git init
git status ফাইলের বর্তমান অবস্থা (Tracked/Untracked) দেখতে git status
git add . সব পরিবর্তন Staging Area-তে যোগ করতে git add .
git add <file> নির্দিষ্ট কোনো ফাইল Staging-এ যোগ করতে git add index.html
git commit -m "msg" স্টেজড ফাইলগুলো রিপোজিটরিতে চিরতরে সেভ করতে git commit -m "feat: navbar"
git log কমিটের ইতিহাস বা পূর্বের কাজের লিস্ট দেখতে git log
git diff ফাইলে কী কী পরিবর্তন করা হয়েছে তা তুলনা করতে git diff
git branch প্রজেক্টের সকল ব্রাঞ্চের নাম দেখতে git branch
git branch <name> নতুন কোনো ব্রাঞ্চ তৈরি করতে git branch feature-login
git switch <branch> একটি ব্রাঞ্চ থেকে অন্য ব্রাঞ্চে পরিবর্তন করতে git switch feature-login
git merge <branch> অন্য কোনো ব্রাঞ্চকে বর্তমান ব্রাঞ্চের সাথে যুক্ত করতে git merge feature-login
git remote -v যুক্ত রিমোট রিপোজিটরির লিস্ট দেখতে git remote -v
git push -u origin main প্রথমবার রিমোটে কোড আপলোড এবং ট্র্যাকিং সেট করতে git push -u origin main
git pull গিটহাব থেকে লেটেস্ট কোড লোকাল প্রজেক্টে টেনে আনতে git pull
git clone <url> অনলাইনের কোনো প্রজেক্ট নিজের কম্পিউটারে ডাউনলোড করতে git clone https://github.com/...

কাজের টিপস:

প্রতিবার কাজ করার শুরুতে এবং মাঝে মাঝে git status দিয়ে দেখে নিন ফাইলের বর্তমান অবস্থা কী। এতে ভুল এড়ানো সহজ হয়!

টপিক ০৭

Branch (ব্রাঞ্চ) কী ও প্রয়োজনীয় কমান্ডসমূহ

Branch (ব্রাঞ্চ) মানে হলো আলাদা একটি কাজের লাইন! প্রজেক্টের মূল কোড (main বা master branch) নষ্ট না করে স্বাধীনভাবে কোনো নতুন ফিচার বা আইডিয়া নিয়ে কাজ করার সুবিধাকে ব্রাঞ্চিং বলা হয়।

main branch
feature-1 branch

ব্রাঞ্চিং-এর দরকার কেন?

  • আপনি যখন নতুন কোনো ফিচার তৈরি করছেন, তখন প্রজেক্টের মেইন কোড সচল থাকে।
  • একই প্রজেক্টে একাধিক ডেভেলপার আলাদা আলাদা ব্রাঞ্চ তৈরি করে একে অপরের কাজে ইন্টারফেয়ার না করে কাজ করতে পারেন।

গুরুত্বপূর্ণ কমান্ডসমূহ:

১. নতুন ব্রাঞ্চ তৈরি করা:

Terminal 
git branch <branch-name>

২. ব্রাঞ্চে মুভ বা পরিবর্তন করা:

Terminal 
git switch <branch-name>  # অথবা git checkout <branch-name>

৩. সকল ব্রাঞ্চের তালিকা দেখা:

Terminal 
git branch

৪. অন্য ব্রাঞ্চকে নিজের ব্রাঞ্চের সাথে মার্জ বা যুক্ত করা:

Terminal 
git merge <branch-name>
টপিক ০৮

Merge & Conflict (মার্জ ও কনফ্লিক্ট)

কোড মার্জিং চমৎকার হলেও মাঝে মাঝে গিটের কাজে কনফ্লিক্ট দেখা দেয়। এটি ডেভেলপমেন্টের একটি অত্যন্ত স্বাভাবিক অংশ।

কনফ্লিক্ট কখন হয়?

যখন দুইজন প্রোগ্রামার একই ফাইলের একই লাইনে আলাদা আলাদা কোড পরিবর্তন করেন, অথবা একজন একটি ফাইল ডিলেট করে দেন আর অন্যজন ওই ফাইলটি এডিট করেন, তখন গিট নিজে থেকে সিদ্ধান্ত নিতে পারে না যে কার কোডটি রাখা হবে। গিট তখন মার্জিং থামিয়ে দেয় এবং একটি Conflict তৈরি করে।

<<<<<<< HEAD (Current Change)
const apiURL = "https://api.mylocalhost.dev";
=======
const apiURL = "https://api.productionserver.com";
>>>>>>> feature-api (Incoming Change)

কনফ্লিক্ট সমাধানের উপায়:

  1. VS Code বা এডিটরে ফাইলটি ওপেন করলে উপরের ছবির মতো অপশন দেখতে পাবেন।
  2. সেখানে ৩টি অপশন থাকে:
    • Accept Current Change: নিজের ব্রাঞ্চের কোডটি রাখতে।
    • Accept Incoming Change: অন্য ব্রাঞ্চ থেকে আসা নতুন কোডটি রাখতে।
    • Accept Both Changes: দুটি কোডই একসাথে রাখতে।
  3. পছন্দমতো অপশন ক্লিক করে ফাইলটি সেভ করুন।
  4. তারপর আবার স্বাভাবিকভাবে ফাইলটি স্টেজিং ও কমিট করুন:
    Terminal 
    git add .
git commit -m "Resolved merge conflict"
টপিক ০৯

Fork, Clone এবং Pull Request

ওপেন সোর্স প্রজেক্টে বা কোনো টিমের সাথে কাজ করার ক্ষেত্রে এই তিনটি শব্দ বা ধারণা বহুল ব্যবহূত হয়।

Fork (ফোর্ক)

অন্য কোনো ডেভেলপারের গিটহাব রিপোজিটরি সম্পূর্ণ কপি করে আপনার নিজের গিটহাব অ্যাকাউন্টে একটি কপি তৈরি করা। এটি সম্পূর্ণ অনলাইন প্রসেস।

Clone (ক্লোন)

গিটহাবের কোনো অনলাইন রিপোজিটরিকে আপনার কম্পিউটারে (লোকালে) ডাউনলোড বা নিয়ে আসা। এর ফলে লোকাল কম্পিউটারে কাজ করা যায়।

git clone <repo-url>

Pull Request (PR)

ফোর্ক বা ক্লোন করা প্রজেক্টে আপনি কোনো কাজ শেষ করার পর, মূল রিপোজিটরির ওনার বা মেইন্টেইনারের কাছে আপনার করা কোডগুলো উনার মূল প্রজেক্টে যুক্ত (Merge) করার জন্য রিকোয়েস্ট পাঠানো।

টপিক ১১

সাধারণ কিছু ভুল ও সমাধান (Common Errors)

গিট ব্যবহার শুরু করার পর নতুনদের সাধারণ কিছু ভুল বা এরর দেখা দেয়। নিচে প্রধান কয়েকটি সমস্যা ও তাদের সহজ সমাধান দেওয়া হলো:

fatal: not a git repository

কারণ: ফোল্ডারটি গিট দ্বারা ট্র্যাকিং করা হচ্ছে না অথবা আপনি ভুল ফোল্ডারে আছেন।

সমাধান: কমান্ডটি রান করুন গিট ইনিশিয়ালাইজ করতে:

git init
fatal: failed to push some refs

কারণ: রিমোট সার্ভারে (GitHub) নতুন কোনো আপডেট আছে যা আপনার লোকাল কম্পিউটারে নেই।

সমাধান: আগে রিমোটের পরিবর্তনগুলো টেনে আনুন (Pull), তারপর পুশ করুন:

git pull
# তারপর
git push
fatal: remote origin already exists

কারণ: আপনার লোকাল প্রজেক্টে ইতিমধ্যে অন্য একটি রিমোট অরিজিনাল লিঙ্ক যুক্ত করা আছে।

সমাধান: রিমোট অরিজিন এর URL টি আপডেট করার জন্য নিচের কমান্ডটি দিন:

git remote set-url origin <new-github-url>
Permission denied (publickey)

কারণ: গিটহাবের সাথে আপনার কম্পিউটার বা টার্মিনালের অথেন্টিকেশন সঠিক নয়।

সমাধান: আপনার গিটহাব একাউন্টে পুনরায় লগইন করুন অথবা SSH Key সেটআপ করুন।

টিপস:

টার্মিনালের এরর মেসেজটি ভালো করে পড়ুন। বেশিরভাগ গিট এররের নিচে সমাধান কী হতে পারে তার একটি গাইডলাইন দেওয়াই থাকে!

টপিক ১২

GitHub Pages-এ প্রজেক্ট লাইভ পাবলিশ করা

GitHub-এর একটি চমৎকার সুবিধা হলো GitHub Pages। এর মাধ্যমে আপনি যেকোনো স্ট্যাটিক (HTML, CSS & JS) প্রজেক্ট একদম ফ্রিতে ইন্টারনেটে লাইভ হোস্ট করতে পারেন।

ধাপসমূহ:

  • প্রধান শর্ত: আপনার প্রজেক্টের রুট ডিরেক্টরিতে একটি index.html ফাইল থাকতে হবে।
  • আপনার কোডগুলো গিটের মাধ্যমে গিটহাবে পুশ করে দিন।
  • গিটহাবে আপনার প্রজেক্টের রিপোজিটরিতে প্রবেশ করুন এবং ডানদিকের Settings অপশনে ক্লিক করুন।
  • বামদিকের মেনুবার থেকে Pages ট্যাবে ক্লিক করুন।
  • Build and deployment সেকশনে:
    • Source: Deploy from a branch সিলেক্ট করুন।
    • Branch: main এবং ফোল্ডার /(root) সিলেক্ট করুন।
  • সবশেষে Save বাটনে ক্লিক করুন।
  • ১-২ মিনিট পর রিফ্রেশ করলে পেজের উপরে আপনার প্রজেক্টের লাইভ লিঙ্কটি দেখতে পাবেন (যেমন: https://username.github.io/reponame)।

অভিনন্দন!

এখন আপনার তৈরি করা ওয়েবসাইটটি পৃথিবীর যেকোনো প্রান্ত থেকে ইন্টারনেটের মাধ্যমে দেখা যাবে।

টপিক ১৩

.gitignore কী এবং এর ব্যবহার

প্রজেক্টের কাজের সময় এমন অনেক ফাইল তৈরি হয় যা আমরা গিটহাবে আপলোড করতে চাই না। যেমন: সিক্রেট পাসওয়ার্ড ফাইল (.env), থার্ড পার্টি মডিউল (node_modules/) অথবা লোকাল ক্যাশ ফাইল। এই ফাইলগুলোকে এড়াতে গিট প্রজেক্টে .gitignore ব্যবহার করা হয়।

কীভাবে ব্যবহার করবেন?

প্রজেক্টের রুট ফোল্ডারে .gitignore নামে একটি ফাইল তৈরি করুন (ফাইলের নামের আগে ডট দিতে ভুলবেন না)। তারপর যে যে ফাইল বা ফোল্ডার বাদ দিতে চান, সেগুলোর নাম ফাইলের ভেতর প্রতি লাইনে একটি করে লিখে সেভ করুন।

একটি আদর্শ .gitignore ফাইলের উদাহরণ:

.gitignore 
# dependencies (লাইব্রেরি)
node_modules/

# local env variables (গোপন চাবি ও পাসওয়ার্ড)
.env
.env.local

# production build (তৈরি হওয়া কোড)
dist/
build/

# system files (কম্পিউটারের ফাইল)
.DS_Store
Thumbs.db

# log files (লগ ফাইল)
*.log

গুরুত্বপূর্ণ সতর্কতা:

কখনো আপনার API Key, Database Password বা .env ফাইল গিটহাবে পুশ করবেন না। এটি আপনার প্রজেক্টের সিকিউরিটির জন্য অত্যন্ত ঝুঁকিপূর্ণ।
